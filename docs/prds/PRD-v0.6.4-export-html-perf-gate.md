# ULog v0.6.4 — `export-html` performance gate (SC1 baseline)

> Mechanically enforces NFR-PERF-60 / SC1 of PRD-v0.6: the static
> HTML export of a 100K-record archive completes in ≤ 30 s on
> GitHub Actions ubuntu-latest CPython 3.12. Closes Story 8.14 of
> the Epic 8 backlog.

---

## 0. 30-second pitch

`ulog export-html` ships with `--max-records 1_000_000` and the
spec says 100K records should export in ≤ 30 s. We have no test
that catches it if a refactor doubles that.

v0.6.4 ships:

1. A reproducible 100K-record fixture (`tests/fixtures/bench_100k.sqlite`,
   generated by `scripts/seed_bench_fixture.py`, NOT committed —
   too big).
2. `tests/bench_export_html.py` using `pytest-benchmark` (already
   in `[dev]` per Story 7.10).
3. A GitHub Actions step that runs the bench, parses
   `--benchmark-json`, and **fails the PR if the median wall time
   exceeds 30 s** (advisory for first 2 runs per Decision E3 of
   PRD-v0.5).
4. The same harness measures **output size** — `--separate-data`
   ≤ 10 MB total, `--inline-data` ≤ 50 MB total (SC3 of PRD-v0.6).

Same advisory-window pattern as the existing v0.5 benchmark step
(`tests/bench_verify.py` / `bench_correlate.py` / `bench_multitrack.py`).

---

## 1. Vision

### 1.1 Why this exists

One observation: **performance regressions in batch exporters are
silent**. A 100K-record archive that took 22 s on v0.6.0 could
take 41 s on v0.6.3 without anyone noticing — the export still
*completes*, it's just slow. By the time a user files an issue
("hey, this is way slower than yesterday"), the regression has
shipped to two releases.

v0.6.4 attacks the silence: a CI job runs the bench on every PR,
and the PR comment surfaces the wall time. Anything > 30 s fails
the build after the 2-run advisory window stabilises.

### 1.2 What v0.6.4 isn't

- **Not a profiler.** The bench measures wall time + output size.
  Memory profiling is out of scope (Story 6.5's RSS ≤ 200 MB
  threshold is checked separately by `tests/test_stability_e2e.py`).
- **Not a per-feature breakdown.** We don't sub-time "filter
  parsing", "render pages", "write JSON sidecars" individually.
  If the overall 30 s budget breaks, we then sub-profile manually.
- **Not a comparison vs other tools** (Splunk export, Datadog
  CSV download, etc.). We benchmark against ourselves; cross-tool
  is a marketing post, not a CI gate.

### 1.3 Target users

- **Release manager** (you) — needs the bench number to publish in
  RELEASE_NOTES.md.
- **Solo dev contributors** — want fast feedback when their PR
  affects export perf.
- **Future-self** — needs a tripwire for regressions during
  v0.7 / v0.8 work that touches `ulog/web/export/`.

### 1.4 Success criteria

| ID | Metric | Target |
|---|---|---|
| SC1 | `pytest tests/bench_export_html.py -m slow --benchmark-only --benchmark-min-rounds=5` runs on a 100K-record fixture | yes |
| SC2 | Median wall time of 5 runs ≤ 30 s on GitHub Actions ubuntu-latest CPython 3.12 | yes |
| SC3 | `--separate-data` total output size ≤ 10 MB; `--inline-data` ≤ 50 MB | yes |
| SC4 | Benchmark step runs in CI with `--benchmark-json benchmark.json`; the file is uploaded as an artifact for trend analysis | yes |
| SC5 | Advisory for first 2 v0.6.4 CI runs (matches Decision E3); strict gate after baseline stabilises | yes |
| SC6 | Fixture seeding takes ≤ 60 s; not committed (regen on demand via `make bench-fixture`) | yes |
| SC7 | Local dev runs the bench in ≤ 1 min on a 2026-spec laptop | yes |

## 2. Scope (v0.6.4)

### 2.1 In scope (~ 150 LOC test + Makefile + CI)

1. **`scripts/seed_bench_fixture.py`** — generates a 100K-record
   chain-mode SQLite DB at `tests/fixtures/bench_100k.sqlite`.
   Mix: 70% INFO, 15% DEBUG, 10% WARNING, 4% ERROR, 1% CRITICAL.
   100 distinct loggers (sectors); 1000 unique tenant_ids in
   context.
2. **`tests/bench_export_html.py`** — uses the `benchmark` fixture
   from `pytest-benchmark`. Three test cases:
   - `test_export_inline_100k` — `--inline-data` mode.
   - `test_export_separate_100k` — `--separate-data` mode.
   - `test_export_filtered_100k` — `--filter "level=ERROR"`,
     should be much faster (post-filter).
3. **`@pytest.mark.benchmark(min_rounds=5, warmup=False)`** on each
   test so we get the median-of-5 measurement.
4. **Output size assertions** at the end of each test:
   `assert sum(p.stat().st_size for p in out.rglob("*")) <
   {10,50}_000_000`.
5. **`make bench-fixture`** — runs the seeder; idempotent (skips if
   the SQLite already exists and is non-empty).
6. **`make bench-export`** — runs `pytest tests/bench_export_html.py
   --benchmark-only` locally.
7. **CI step** in `.github/workflows/ci.yml`:
   ```yaml
   bench-export:
     needs: test
     runs-on: ubuntu-latest
     continue-on-error: true  # advisory for first 2 runs
     steps:
       - uses: actions/checkout@v4
       - run: pip install -e ".[dev,storage,web]"
       - run: make bench-fixture
       - run: pytest tests/bench_export_html.py -m slow --benchmark-only \
              --benchmark-json benchmark.json
       - uses: actions/upload-artifact@v4
         with:
           name: bench-results
           path: benchmark.json
   ```
8. **BENCHMARK.md update** — populate the `_tbd_` row from
   `tests/bench_export_html.py::test_export_separate_100k` once the
   first CI run completes.

### 2.2 Explicit non-goals

- **Profiling integration (py-spy, cProfile, austin)** — out. We
  measure totals; sub-profiling is on-demand, not a CI gate.
- **Cross-OS benchmark matrix (Windows / macOS runners)** — out.
  Linux as the single reference per Decision E3 of PRD-v0.5.
- **Comparison with a previous run automatically commenting on the
  PR** — out for v0.6.4. The artifact upload lets users diff
  manually. v0.6.5 (or v0.8) could wire `pytest-benchmark`'s
  `--benchmark-compare` to a PR comment.
- **Strict gate at first run** — by Decision E3, advisory mode for
  the first 2 v0.6.4 CI runs to gather baseline noise, then a
  follow-up PR flips to strict.

### 2.3 Edge cases & failure modes

| Case | Behaviour |
|---|---|
| Fixture missing on first CI run | `make bench-fixture` regenerates; cached via `actions/cache` keyed by the seeder script's hash. |
| Benchmark fails with `pytest_benchmark.fixture.BenchmarkSession`-related import error | `pytest-benchmark` listed in `[dev]` (Story 7.10); CI install picks it up. |
| 100K-record DB seed exceeds `--max-records` | Seeder caps at 100K exactly (assert in script). |
| Runner has noisy neighbours → bench wildly flaky | Median-of-5 absorbs outliers; advisory window also covers it. |
| Output dir survives between bench runs and breaks `--force` checks | `tmp_path_factory` fixture from pytest cleans up per test. |

## 5. Decisions

### D1 — Median of 5 vs of 10

**Median of 5** — pytest-benchmark's default `min_rounds=5` is the
sweet spot. 10 doubles CI time without much noise reduction on a
30 s job (CV stays similar).

Source: <https://pytest-benchmark.readthedocs.io/en/latest/usage.html>

### D2 — `@pytest.mark.benchmark(warmup=False)`

Disable warmup — the export's first run is representative of real
usage (no caches warmed). The harness already runs 5 rounds, so
outlier robustness is handled.

### D3 — Fixture not committed; regenerated by CI

The 100K-record SQLite is ~50 MB — too big to commit cleanly.
Seeder script + `make bench-fixture` rebuild on demand. CI caches
the rebuild output via `actions/cache@v4` keyed by
`hashFiles('scripts/seed_bench_fixture.py')`.

### D4 — `--benchmark-compare-fail` not used (yet)

`--benchmark-compare-fail "median:1.5"` could fail the PR on a
1.5× slowdown vs a saved baseline. Useful but premature — we don't
have a baseline file yet, and saving one to git creates merge
conflicts. v0.7 may revisit.

Source: <https://pytest-benchmark.readthedocs.io/en/latest/comparing.html>

### D5 — `-m slow` marker

The bench is in the `slow` marker pool (already defined in
`pyproject.toml::tool.pytest.ini_options::markers`). `pytest` runs
without `-m slow` skip it by default. CI runs `-m slow
--benchmark-only` explicitly.

## 6. Operating manual

### Local dev

```bash
# Generate the fixture once (~30 s).
make bench-fixture

# Run the bench (~3 minutes for 5 rounds of 30 s each).
make bench-export

# Or run just one variant.
pytest tests/bench_export_html.py::test_export_separate_100k \
    -m slow --benchmark-only -v
```

The `.benchmarks/` dir is created (gitignored). Inspect:

```bash
ls .benchmarks/Linux-CPython-3.12-64bit/
# 0001_*.json
```

### CI

After v0.6.4, the workflow gains:

```yaml
bench-export:
  needs: test
  runs-on: ubuntu-latest
  continue-on-error: true  # remove after baseline stabilises
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with: { python-version: "3.12" }
    - name: Cache bench fixture
      uses: actions/cache@v4
      with:
        path: tests/fixtures/bench_100k.sqlite
        key: bench-fixture-${{ hashFiles('scripts/seed_bench_fixture.py') }}
    - run: pip install -e ".[dev,storage,web]"
    - run: make bench-fixture
    - run: |
        pytest tests/bench_export_html.py \
          -m slow --benchmark-only \
          --benchmark-min-rounds=5 \
          --benchmark-json=benchmark.json
    - name: Check 30s gate
      run: |
        python -c "
        import json
        b = json.load(open('benchmark.json'))['benchmarks']
        sep = next(x for x in b if 'separate' in x['name'])
        median_s = sep['stats']['median']
        if median_s > 30.0:
            print(f'::error::Export bench median {median_s:.1f}s exceeds 30s budget')
            exit(1)
        print(f'OK: median {median_s:.1f}s')
        "
    - uses: actions/upload-artifact@v4
      with:
        name: bench-results
        path: benchmark.json
```

### Reading the JSON output

```python
import json
data = json.load(open("benchmark.json"))
for b in data["benchmarks"]:
    print(f"{b['name']}: median={b['stats']['median']:.2f}s "
          f"(min={b['stats']['min']:.2f}s, max={b['stats']['max']:.2f}s)")
```

Source: <https://pytest-benchmark.readthedocs.io/en/latest/usage.html>

### Promoting from advisory to strict

After 2 CI runs land cleanly on `main` with stable median (delta
< 10 %):

1. Remove `continue-on-error: true` from `.github/workflows/ci.yml`.
2. The `Check 30s gate` step now fails the build on regression.
3. Commit with `ci: harden export-html perf gate (post-advisory)`.

## 7. Open questions

- **Q1** : Should we also benchmark **incremental exports** (run
  twice, second run from a warm SQLite page cache)? The 30 s
  budget is cold-cache; warm runs may surprise. Lean no for
  v0.6.4 — single budget is simpler.
- **Q2** : `--benchmark-compare-fail "median:1.5"` for trend
  regression detection. Worth a v0.7 PRD when we have a stable
  baseline file in the repo. Trade-off : baseline merge-conflicts
  on every release branch.
- **Q3** : Add a "memory" axis (peak RSS during export)?
  `tracemalloc` could measure it without external deps. Defer to
  v0.7.
