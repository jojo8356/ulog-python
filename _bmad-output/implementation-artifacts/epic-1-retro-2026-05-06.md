## 2. What went well

### 2.1 Disciplined CS → VS → DS → CR cycle

Every story ran the same loop. The validation step (VS) consistently caught spec gaps before implementation:
- Story 1.3: 7 enhancements applied pre-DS (test count math, `tests/` scope clause, parametrize edges, helper-importability test).
- Story 1.6: 8 fixes pre-DS (lucide naming, baseline 20 not 122, slowest_only ORDER BY concrete code, `<details open>` UX).
- Story 1.7: 8 fixes pre-DS (keep() signature pin, page-drop in qs_minus_test_id, data-attribute over Tailwind class, parametrize URL encoding test).
- Story 1.10: 4 fixes pre-DS including the WAL reentrancy bug (C1) and Linux mountinfo root-mount bug (C2) — both real correctness defects caught before code was written.

Pre-DS validation prevented an estimated ~20 round-trips of "implement → CR finds bug → fix" by surfacing the issues in the spec.

### 2.2 3-reviewer adversarial CR consistently surfaced real defects

Running Blind Hunter + Edge Case Hunter + Acceptance Auditor in parallel (Sonnet 4.6) on every story produced ~6-12 findings per story. Triage discipline (PATCH / DEFER / DISMISS-with-rationale) kept the noise/signal ratio honest. Highlights:
- Story 1.7 EH1 (ghost-count `test_id` poisoning) — PRD-v0.2.1 contract violation; would have shipped silently otherwise.
- Story 1.9 EH2 (nested `test_event` outer-id destruction) — switched from bind/unbind to `ulog.context()`; real correctness bug.
- Story 1.10 C1 (WAL helper engine disposal during open connection) — would have caused intermittent SQLAlchemy pool errors under xdist.

### 2.3 Test coverage built incrementally and stayed honest

92 net new tests, no flakiness reported. Recurring discipline patterns that paid off:
- Anchor record-count assertions with `==`, never `>=` (Story 1.2/1.3 P1/P3/P4/P6 lineage carried through).
- Set-comparison on multi-value query params (Story 1.8 P3).
- `data-` attributes for stable test hooks instead of Tailwind class strings (Story 1.7 P1).
- Literal-equality assertions for nodeid format stability — pytest's nodeid is deterministic by construction, so locking the literal IS the stability proof (Story 1.3 design decision).

### 2.4 Frozen invariants held across all 11 stories

Zero modifications to `ulog/__init__.py`, `setup.py`, `context.py`, `formatters.py`, `_color.py`, `handlers/`. Story 1.4 was 100% tests-only; Stories 1.1, 1.6-1.11 each touched a tightly-scoped file set. The architectural commitment to additive-only sub-package layout (`ulog/testing/` / `ulog/web/viewer/`) made this straightforward.

### 2.5 Autonomous execution with periodic commits

Following Johan's "ne plus me poser de questions" directive, the cycle ran unsupervised with logical commits at natural boundaries (every 2-3 stories). 8 commits during this session, all self-contained, each with detailed messages tracking which CR patches were applied and dismissed.

## 3. What didn't go well

### 3.1 Pytest auto-discovery footgun on `test_event`

Story 1.9's `test_event` function (pytest-pattern name) was auto-collected as a top-level test in any module that did `from ulog.testing import test_event`. Caught in DS (10/11 pass + 1 spurious "ERROR at setup of test_event"), fixed via `test_event.__test__ = False`. Cost: ~30 minutes of debugging.

**Lesson:** any `ulog/testing/`-exported function whose name matches `test_*` MUST set `__test__ = False`. Document in the testing sub-package's `__init__.py` README convention.

### 3.2 mypy regressions slipped past Stories 1.9 / 1.10 DS gates

Both stories' "mypy clean" claim was true at the moment of writing the DS notes, but a later `python -m mypy ulog/testing/` run surfaced 4 errors I'd introduced:
- `test_event.py`: missing return type on `_ulog()` helper + 2 untyped-call errors.
- `pytest_plugin.py`: `kernel32.GetDriveTypeW` returns `Any` (ctypes); needed explicit `bool()` cast.

Caught and fixed before the final commit, but the DS-step "mypy clean" gate didn't catch it because I ran `mypy ulog/testing/` AFTER changes were made but BEFORE the new helpers were stress-tested.

**Lesson:** run `mypy` AFTER the test suite passes, not BEFORE. Add to the DS step's verify checklist explicitly.

### 3.3 Cumulative-diff confusion in Story 1.8 CR

The Acceptance Auditor for Story 1.8 flagged AC8 as PARTIAL because the cumulative `git diff HEAD` showed `list.html` modified — but those changes belonged to Stories 1.6/1.7 (uncommitted at the time). The auditor was reading a CUMULATIVE diff, not a Story 1.8 delta. False alarm dismissed with rationale, but it's a recurring pattern.

**Lesson:** when committing per-2-3-stories, the cumulative-diff confusion will keep happening on adversarial reviews. Either commit per-story (heavier ceremony) or explicitly tell reviewers "review only the Story X delta within this cumulative diff."

### 3.4 Test fixture coupling between stories

Story 1.7's `_make_test_records_with_app_logs` fixture was reused by Story 1.8 tests. Story 1.8 inherited the fixture's exact shape (3 records per test). If Story 1.7 ever changes the fixture's record count, Story 1.8 tests break implicitly. The Edge Case Hunter for Story 1.8 flagged this as LOW.

**Lesson:** test fixtures shared across stories should be locked (record count, schema) with a docstring contract. Or each story should define its own fixture and accept the duplication.

### 3.5 Settings cache bug only surfaced in Story 1.6

`_make_django_client` was a pre-existing helper that worked for the original 20 web tests because they all used the same `sqlite_fixture` path. Story 1.6's tests use distinct DB paths per test, exposing that `settings.ULOG_LOGS_PATH` is read once at `django.setup()` and never refreshed. Fixed in Story 1.6's DS step (force-update `settings.ULOG_LOGS_PATH` from the env var on every `_make_django_client` call).

**Lesson:** pre-existing helpers should be audited for "works because of accidental coincidence" patterns BEFORE the first story that violates the coincidence. Could have been caught earlier if Story 1.5 had run a tests/ smoke that varied DB paths.

### 3.6 Story 1.9 spec said "10 tests" but delivered 13

CR caught the count mismatch (Auditor + Edge Hunter both flagged it). The spec's "10" was a clean-architecture estimate; CR added 3 regression tests for AC4 exception+explicit-outcome / nested test_event / empty-name guard. Final count 13 is better, not worse — but the DoD claim "10 tests" was wrong by the time it shipped.

**Lesson:** test-count claims in specs should be ranges or "≥N". Or update them post-CR if patches add tests.

## 4. Key insights and lessons learned

### 4.1 Spec-driven validation is the highest-leverage step

The VS (validation) step caught more real defects than the CR step in 6 of 11 stories. The pattern: a fresh-context independent reviewer reads the spec without the implementation context I had, and surfaces ambiguities or wrong assumptions BEFORE code is written. This is cheaper than catching the same defects in CR (where the bug already exists in tests + code + docs).

**Apply to Epic 2:** invest more in VS-step depth. Have the VS reviewer trace the AC carefully against the source PRD, not just the spec.

### 4.2 Adversarial CR finds bugs the original author cannot

3 reviewers in parallel found ~50 patches across the epic. Of those, 6 were genuine correctness defects (vs cosmetic / style). The author who wrote the code consistently MISSES the same class of bugs (encapsulation issues, edge-case scoping, fixture coupling). The reviewer-as-fresh-eyes is structurally better at this than the author re-reading their own work.

**Apply to Epic 2:** do not skip CR even on "small" stories. The two smallest stories (1.4 pure-tests; 1.11 markdown + 1 dict line) each surfaced 2-3 real findings.

### 4.3 Patches > rewrites

Every CR patch was a tiny diff (1-30 lines) targeting a specific finding. The discipline of "minimum viable fix" kept review cycles short and the diff reviewable. A larger refactor would have invalidated the prior testing.

**Apply to Epic 2:** when CR surfaces a finding, the first instinct should be "what's the smallest patch that closes this?" not "let me rewrite the helper."

### 4.4 Test fixtures are first-class architecture

Story 1.6's `_make_test_records_db`, Story 1.7's `_make_test_records_with_app_logs`, Story 1.10's monkeypatch patterns — these are reused across stories and become contracts. They should be documented as carefully as production code.

**Apply to Epic 2:** when a story creates a test fixture that another story will reuse, name it explicitly and add a docstring locking the contract.

### 4.5 Permissions evolve to keep up with autonomy

Started Epic 1 with ~35 narrow permissions. Ended at 48 broad allow patterns + 6 deny (git push). Each permission narrowing or deny was a deliberate choice. The defaultMode shift from `default` to `acceptEdits` was the biggest single accelerant — eliminates per-edit prompts that used to interrupt every 30 seconds.

**Apply to Epic 2:** keep `acceptEdits` + `Bash` allow + `git push` deny. The pattern works.

### 4.6 `ulog.context()` over `bind`/`unbind` for nestable scopes

Story 1.9's nested `test_event` bug surfaced this pattern: `bind`/`unbind` is fine for non-nestable surfaces (Story 1.2's pytest hookwrapper), but anything that COULD nest needs `ulog.context()` (ContextVar token-based). The `bind` keypath destroys outer context on inner unbind.

**Apply to Epic 2:** any new public API in `ulog/testing/` or `ulog/web/` that scopes context should default to `ulog.context()`.

## 5. Significant discoveries — Epic 2 impact

### 5.1 No fundamental Epic 2 plan changes required

Epic 2 (v0.4 Author attribution) is architecturally independent of Epic 1. The epics.md notes: "Standalone: independent of Epic 1 conceptually. Composes with Epic 1 if both extras are installed." Verified: nothing in Epic 1's discoveries invalidates Epic 2's plan.

### 5.2 Patterns to carry forward into Epic 2

- **Adapter extension protocol** — Story 1.6/1.7/1.8 added new fields to `Filters` / `QueryResult` and new helpers on `SQLiteAdapter`/`JSONLAdapter`/`CSVAdapter`. Epic 2's `AuthorIndex` will follow the same pattern: extend the adapter with `count_records_for_author()` / `get_author_for_record()` parallel to Story 1.8's `count_records_for_test_id` / `get_test_summary_row`.
- **Sidecar SQLite for non-SQLite inputs** — Epic 2 §Implementation notes mentions `<logs>.authors.sqlite` sidecar for JSONL/CSV inputs (Decision A3). Story 1.10's JSONL fallback (`<path>.sqlite` → `<path>.jsonl`) established the path-stem-swap convention. Reuse the same naming pattern.
- **Subprocess-only git, no GitPython** — Epic 2 §Implementation notes locks this in. Aligns with NFR-DEP-50. Story 1.10's macOS `mount` subprocess (with timeout, capture_output, defensive `subprocess.SubprocessError` catch) is a template for how to invoke `git blame --porcelain` safely.

### 5.3 Risks visible from Epic 1 → Epic 2

- **`AuthorIndex` performance under large repos**: Story 1.8's `_build_test_summary` is O(N) over all outcome records on every detail-view render (TODO(v0.4 NFR-PERF) flagged). Epic 2's `AuthorIndex` will face the same problem at larger scale (`git blame` on a 100k-line file takes seconds). Plan for caching from day one — don't make Epic 2's analog of Story 1.8's `get_test_summary_row` an O(N) scan per request.
- **`/diff/<sha>` view security** (NFR-SEC-30 / Decision D4): Story 1.10's URL/path validation patterns (sqlite path → JSONL path stem swap with `rsplit('.', 1)[0]`) is good prior art. The `/diff/<sha>` endpoint must validate `sha` matches `^[a-f0-9]{40}$` before passing to `git show`; never pass user input to subprocess directly.
