## What went well

1. **The `_cli/` package convention scaled cleanly across 3 subcommands.** `cmd_verify`, `cmd_repair`, `cmd_purge` each follow the same `register(subparsers)` + `run(args)` shape declared in `architecture.md` Decision C1. Adding a 4th subcommand would be ~30 minutes of mechanical work — no scaffolding redesign needed.

2. **Shared `_parse_ts` + `sha256_record` + `canonical_record_json` between writer and verifier prevented drift.** The bug discovered in Story 3.7 (SQLite text() returns ts as space-separated string, not datetime) was localized to one helper and fixed once; verify, repair, and any future re-walker share the same parsing path.

3. **Sentinel attribute pattern on the SQLAlchemy engine (`_ulog_chain_begin_immediate`).** Mirrors the existing `_ulog_managed` flag on handlers (`tests/test_handlers.py:27`). Allowed multiple `SQLiteChainWriter` instances on the same engine without stacking BEGIN IMMEDIATE listeners. Verified by `test_sqlite_chain_writer_begin_immediate_registered_once`.

4. **Story 3.2 trigger composition into Story 3.6's `immutable_when` flow worked end-to-end first-try.** `test_immutable_trigger_blocks_update_on_immutable_when_marked_row` confirms: `immutable_when` sets the flag at INSERT, the trigger blocks UPDATE on that row. Zero coupling between the two stories at the code level — both just read/write the `immutable` column.

5. **Dialect gating from day 1.** Both Story 3.2 (triggers) and Story 3.5 (WAL pragma) check `self._engine.dialect.name == "sqlite"` and silent-return otherwise. v0.7 Postgres can layer its own `ChainWriter` impl without touching these guards (Decision B3).

6. **`@runtime_checkable Protocol` enabled mock-friendly chain tests.** `MagicMock(spec=ChainWriter)` passes `isinstance` (test `test_chain_writer_protocol_signature`). Future chain-related unit tests can mock the writer without a SQLite DB — directly supports AC3 of Story 3.4.

7. **Retrospective bug surface from real stress test in Story 3.11.** The single-process 4-thread concurrent test in Story 3.4 passed cleanly, but the 8-proc × 10K stress in 3.11 broke at record #272 because `get_last_hash + append` were two separate transactions. Fix: `append_atomic` that does prev-read + hash-compute + INSERT in ONE `BEGIN IMMEDIATE` txn. **Caught BEFORE shipping.**

## What could be better

1. **The fail-safe direction for `immutable_when`-raises was wrong in my Story 3.6 spec.** I wrote `immutable=0` (fail open), but the architecture (Decision B5 at `architecture.md:686-691`) says `immutable=1` (fail safe — preserve forensic evidence). Story 3.12 caught and corrected this, but the wrong wording propagated into Story 3.6's spec doc, its impl, its tests, and its completion notes — three files needed re-touching. **Action for next epic:** when an AC references "Decision Bx" in epics.md, re-read the architecture decision verbatim before writing the spec, instead of paraphrasing from memory.

2. **Raw `text()` SELECT bypasses SQLAlchemy's column-type adapters.** Discovered while writing Story 3.7's verify recompute — `ts` came back as a string from `text("SELECT ts ...")`, not a `datetime`. Same root cause for JSON columns coming back as strings. Added `_parse_ts()` + `json.loads(...)` round-trip helpers in verify, repair, and the test recompute path. Pattern is now documented inline. **Action for v0.6+ if we add new SQL types:** prefer SQLAlchemy Core (`Table.select()`) for chain-state queries to keep type adapters in the loop, OR document the round-trip pattern as the v0.5 contract.

3. **Story 3.5's `test_chain_emit_record_hash_matches_canonical_recompute` was scope-cut.** The brittle SQLite-round-trip recompute test was dropped because reconstructing the exact dict at verify-time from the DB form is the literal job of Story 3.7's verify walk. Could have caught the `ts` round-trip bug 2 stories earlier. **Action:** when a test requires "reconstruct emit-time state from storage", reach for the verify walk (or its precursor logic) instead of inventing a one-off comparison.

4. **Single 91s slow test pulls the marker bookkeeping for one use case only.** The `slow` marker registration in `pyproject.toml` is now there. As more long stress tests land (Epic 4 replay, Epic 6 frontend), the slow tier may need a CI matrix entry rather than just an opt-in flag. **Action for Epic 7 (release consolidation):** explicit `pytest -m "not slow"` (default fast) + `pytest -m slow` (release gate) split documented in `RELEASE_CHECKLIST.md`.

5. **The cross-process race in `_chain_append` (get-then-append as 2 txns) was not caught by single-process concurrency tests.** Story 3.4's `test_sqlite_chain_writer_concurrent_append_serialised` (2 threads × 50 appends, 100 rows) passed because both threads share the same lock + the same engine — `engine.begin()` serialises them within the process. The bug only emerged under 8 *separate processes* (no shared lock, only SQLite file lock). **Lesson:** when an AC says "multi-process", the test MUST use subprocesses. Threads-in-one-process are a different concurrency story.

## Lessons captured for Epic 4+

1. **Atomic-write idiom — `tmp + os.replace`.** Used for `verify_state.json` (Story 3.10) and the JSONL sidecar in `cmd_repair`. Same pattern should land in Epic 5 (`incidents` sidecar) and Epic 7 (release `ulog export-html` static output).

2. **Hash-function injection over pre-computed args.** The signature shift from `append(record, record_hash, prev_hash)` (caller-precomputed, racy across procs) to `append_atomic(record, hash_fn)` (compute-inside-txn, race-free) is the canonical pattern. Apply to Epic 4's `replay()` if it ever computes a "session digest" — pass a `digest_fn`, don't pass the digest.

3. **Stderr `print` for fail-safe diagnostics inside the SQL handler.** `warnings.warn` would have been intercepted by `warnings.simplefilter("error")` test setups (deal-breaker if a user has that in conftest). `print(..., file=sys.stderr)` is unconditional and matches Decision B5's "do NOT use ulog logging — recursion". Applied in `_record_to_row`'s `immutable_when` fail-safe branch.

4. **Subprocess-as-the-real-test for cross-process invariants.** `tests/test_chain_concurrency.py` and `tests/test_handlers.py::test_sql_handler_no_race_under_concurrent_bootstrap` (Story 1.13) both spawn `subprocess.run(sys.executable, "-c", script_text, ...)`. Slot this idiom into Epic 6 (cross-service correlate) where the test needs N processes appending records that another process correlates.

5. **`@pytest.mark.slow` for tests over ~5s.** Register in `pyproject.toml` `[tool.pytest.ini_options].markers`. Run gates in CI: fast (`-m "not slow"`) on every PR, slow on release-tag branches.
